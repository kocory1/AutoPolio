# Autofolio (오토폴리오) 기획서

**Slogan:** "From Code to Career" – 증거 기반 개발자 이력서

**문서 버전:** 1.1  
**최종 정리일:** 2026-03-04

---

## 1. 개요

### 1.1 문제 정의 (Problem)

| 구분 | 설명 |
|------|------|
| **Developers hate writing** | 개발자는 코드는 잘 짜지만, 이를 '채용 담당자가 좋아하는 언어(비즈니스 임팩트)'로 번역하는 데 어려움을 겪음. |
| **The "Hallucination" Trap** | 기존 AI(ChatGPT 등)에게 자소서를 맡기면, 없는 경험을 지어내거나(Hallucination) 뜬구름 잡는 미사여구(MSG)만 남발함. |
| **Disconnected Evidence** | 자소서의 주장("성능 50% 개선")과 실제 증거(GitHub Commit/Code)가 분리되어 있어 신뢰도가 낮음. |

### 1.2 솔루션 (Solution)

> "자소서를 쓰는 게 아니라, 내 경험을 문서화(Documentation)한다."

Autofolio는 **GitHub 코드/커밋** + **이력서/포트폴리오 문서**를 수집하여 **검증된 성과(Asset)**로 변환하고, **채용 공고(Target)**에 맞는 증거를 조합해 **신뢰 가능한 자소서 초안** 작성을 돕는 서비스입니다.

## 2. 입력 소스

| 입력 | 포맷 | 처리 방식 |
|------|------|-----------|
| **GitHub** | 코드, 커밋 | GitHub API → 기본 제외 → RAPTOR 임베딩 |
| **이력서** | pdf, ppt | 전처리 (필요 시 멀티모달 LLM) |
| **포트폴리오** | pdf, ppt | 전처리 (필요 시 멀티모달 LLM) |

→ 모든 소스는 **User 프로필 DB**로 통합

## 3. 핵심 기능 (Core Features)

### Phase 1: Data Mining & Portfolio (자산화)

**컨셉:** "진흙 속의 진주 찾기"

| 기능 | 설명 |
|------|------|
| **GitHub 연결** | OAuth/PAT로 연결 → **레포 목록 노출** → **유저가 선택한 레포만** 분석 수행. |
| **이력서/포트폴리오 업로드** | **PDF, PPT** 업로드 → 멀티모달 전처리 → 텍스트/구조 추출 |
| **기본 제외** | node_modules, .git 등 기본 경로/확장자 제외 (RAPTOR가 나머지 정제) |
| **RAPTOR 임베딩** | 폴더 구조 기반 계층적 요약 → Vector DB 저장 |
| **Asset Generation** | 코드/커밋/문서를 분석해 STAR(상황-과제-행동-결과) 구조의 성과 문장 후보로 변환. |
| **포트폴리오 생성** | 분석된 데이터를 바탕으로 웹 포트폴리오 생성. |

### Phase 2: Job Intelligence (공고 분석·적합도)

**컨셉:** "지피지기면 백전백승"

| 기능 | 설명 |
|------|------|
| **채용 공고 파싱** | 채용 공고 URL → **LLM 기반 크롤링** → 담당 업무 / 자격 요건 / 우대 사항 / 기업명 / 기업 인재상 / 포지션명 추출 |
| **자소서 문항 입력** | 유저가 자기소개서 문항 직접 입력 |
| **Job Fit 점수화** | **User DB**(프로필·임베딩) vs **공고 파싱 API 반환값** 비교 → Job Fit 점수 API |
| **문항 기반 에셋 매칭** | 문항과 공고에 맞는 에셋 선별 및 부족 포인트 점검 (draft 내부 처리) |

### Phase 3: Evidence-Based IDE (증거 기반 작성)

**컨셉:** "팩트(Fact)로 글(Draft)을 제압한다"

| 기능 | 설명 |
|------|------|
| **Split-View UI** | 좌측: 근거 카드(Code/Commit/문서), 우측: 에디터(Draft) 배치. |
| **Writer (AI)** | 합격 자소서(유사 샘플) + 에셋(유저 DB) 참고하여 **초안** 작성. |
| **Inspector (AI)** | 작성된 문장에 대해 **보완할 점을 제안**. **실제 수정은 유저가 수행.** |
| **대화형 첨삭** | 유저가 수정 후 Inspector와 **대화를 이어가며 반복 첨삭**하는 플로우. |
| **추후 확장** | Inspector 제안 시 해당 코드/커밋을 UI에 바로 노출하는 기능은 **부가 기능**으로 추후 검토. |

## 4. 기술 아키텍처 (Tech Stack)

| 구분 | 기술 스택 | 역할 |
|------|-----------|------|
| Orchestration | LangGraph | Writer/Inspector 흐름 및 상태(State) 관리 |
| LLM | GPT-4o / Claude 3.5 | 코드 분석, 텍스트 생성, 멀티모달 전처리 |
| Vector DB | Pinecone / Chroma | 코드 청크(Chunk) 임베딩 및 시맨틱 검색 (RAPTOR 구조) |
| Backend | FastAPI (Python) | 비동기 처리 및 API 서버 |
| Frontend | React / Next.js | Split-View UI 및 인터랙티브 에디터 |
| 데이터 수집 | **GitHub API** | 레포 목록, 코드(트리/블롭), 커밋 수집 |
| 채용 공고 파싱 | **LLM 기반 크롤링** | URL 접근 + 구조화 추출 단일 스텝 |
| 합격 자소서 | **잡코리아, 링커리어** | 합격 사례 크롤링 → DB 저장 |

## 5. 데이터 수집

### 5.1 GitHub

- **방식:** GitHub **REST/GraphQL API** 사용.
- **흐름:** 유저 인증 → 레포 목록 조회 → **유저가 선택한 레포만** 분석.
- **기본 제외:** node_modules, .git, __pycache__ 등 (RAPTOR가 상위 레벨에서 자연스럽게 정제)

### 5.2 이력서/포트폴리오

- **포맷:** pdf, ppt
- **처리:** 텍스트 추출, 필요 시 멀티모달 LLM으로 이미지/레이아웃 해석
- **용도:** GitHub에 없는 경력, 비개발 경험, 과거 프로젝트 보완

### 5.3 합격 자소서

- **소스:** 잡코리아 (`/company/{id}/PassAssay`), 링커리어 (`/cover-letter/{id}`)
- **수집:** LLM 기반 크롤링으로 회사/포지션/문항/답변/전문가 피드백 추출
- **용도:** Writer/Inspector 참고

## 6. 제약·범위 정리

| 항목 | 결정 |
|------|------|
| 채용 공고 크롤링/법적 이슈 | 당장은 괜찮다고 간주하고 진행. 필요 시 추후 검토. |
| Inspector의 코드/커밋 UI 노출 | 부가 기능. 추후 확장으로 검토. |
| API Rate Limit | 당장은 고려하지 않음. 구현 시 대응. |
| 합격 자소서 크롤링 | 잡코리아, 링커리어만 (robots.txt 허용 확인됨) |
| **GitHub 연동 해제** | 제공하지 않음. 로그아웃으로 충분. |
